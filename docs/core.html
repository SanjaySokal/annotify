Core framework

The non-decorator pieces — Router, Scanner, Resolver, Server, AppBuilder, and the error model. Read this to understand the request lifecycle or to extend the framework.

Router

src/core/router.ts  ·  src/core/route-node.ts

The router is a trie keyed on URL segments. Each node has:

  • children — a Map of literal-segment children (e.g. "users").
  • varChild — at most one path-variable child (e.g. ":id").
  • handlers — a Map from HTTP method to RouteEntry.

Matching algorithm

For each segment of the request path:

  1. Look for a literal child first. If found, descend.
  2. If no literal child, fall back to varChild. Record the variable name → value mapping.
  3. If neither exists, the route doesn't match → null.

Literal priority is important: /users/me matches before /users/:id when both are registered. Path variables are URL-decoded before being stored.

Method Not Allowed (405)

When a path exists but the request method has no handler, Router.methodsAt(path) returns the list of registered methods. The server emits an Allow header with them and a 405 JSON body.

Resolver

src/core/resolver.ts

The resolver inspects each parameter position on the bound handler, looks up the corresponding ParamMeta from entry.paramTypes[i], and pulls the value from the right place in the RequestContext:

kindSource
'path'ctx.pathVars[name]
'param'ctx.query[name] (first value, with default fallback)
'body'ctx.body
'header'ctx.req.headers[name.toLowerCase()]
'req'ctx.req
'res'ctx.res
'context'ctx
(no meta)Positional: ctx.req, ctx.res, ctx, then undefined

If a parameter has no decorator and is not in position 0–2, undefined is passed. This keeps handlers tolerant of optional/extra parameters.

Scanner

src/core/scanner.ts

The scanner recursively walks a directory looking for compiled .js / .mjs files, dynamically imports each, and collects exports whose constructor has ROUTE_METADATA set.

scanControllers(rootDir):
  files = walk(rootDir)             // recursive fs.readdirSync
  ctors = []
  for file in files:
    if ext not in (.js, .mjs): continue
    url = pathToFileURL(file).href  // Windows-safe file:// URL
    mod = await import(url)
    for export of Object.values(mod):
      if export is function and export[ROUTE_METADATA]:
        ctors.push(export)
  return ctors

Windows note. pathToFileURL from node:url is required because dynamic import() on Windows refuses raw absolute paths. The scanner always uses URLs.

The scanner skips files that fail to import — they're likely non-class helpers — so you can keep utility modules in the same directory.

Server

src/core/server.ts

The server wraps Node's native http.createServer with the request-handling pipeline:

  • Parse the request method and URL.
  • Read the body (only for POST/PUT/PATCH). Enforce 1 MB cap (→ 413). Reject non-JSON content types (→ 415).
  • Call router.match. On miss, fall through to 405 or 404.
  • Build a RequestContext and call resolver.resolveArgs.
  • Invoke the bound handler, await its result, and serialize.
  • Any thrown error is funneled through sendInternalError, which preserves HttpError status codes and turns everything else into 500.

Response conventions

  • Handler returns a value → 200 + JSON body.
  • Handler returns undefined → 204 No Content.
  • Handler throws HttpError(status, message) → that status + JSON { error, message }.
  • Any other throw → 500 + JSON { error: "Internal Server Error", message }.
  • If the handler already wrote to res, the server does not write again.

AppBuilder

src/core/app.ts

class AppBuilder {
  scan(dir: string): Promise<this>        // auto-discover controllers
  register(ctor: Function): this           // manual registration
  setDefaultIsRest(v: boolean): this        // default JSON behavior
  build(): Router                          // instantiate + bind handlers
  listen(port: number, host?: string): Promise<http.Server>
}

Typical usage:

const app = new AppBuilder();
await app.scan('./dist/controllers');
const server = await app.listen(3000);
// server is an http.Server; close() it on shutdown

listen returns the underlying http.Server. Wire it to SIGINT/SIGTERM for graceful shutdown.

Errors

src/core/errors.ts

The HttpError class is the public error type:

class HttpError extends Error {
  status: number;
  details?: unknown;
  constructor(status: number, message: string, details?: unknown);
}

Internal helpers (sendNotFound, sendMethodNotAllowed, sendPayloadTooLarge, sendUnsupportedMediaType, sendInternalError) are not exported; they're used by the server and the resolver. You can still construct an HttpError from any handler and the framework will turn it into the right HTTP response.

Metadata side-channel

src/decorators/metadata.ts

There is no reflect-metadata dependency. Instead, src/decorators/metadata.ts defines a module-level WeakMap<object, Record<string, ParamMeta[]>> keyed by the class prototype. Parameter decorators write to it; method decorators read it. Once a method's metadata is consumed, that method's slot is deleted — the registry is per-decoration lifecycle.

The class-level metadata (basePath, isRest, the array of route entries) lives on the class itself under the symbol Symbol.for('annotify.route'), set via Object.defineProperty with enumerable: false so it does not pollute Object.keys(controller).

Request logger

src/core/logger.ts

Turn on per-request access logging with app.useLogger({ enabled: true }). The default format prints one line per request:

[annotify] 2026-07-06T15:32:18.123Z  GET    /users/1     200  3 ms
[annotify] 2026-07-06T15:32:18.158Z  POST   /users/      201  1 ms
[annotify] 2026-07-06T15:32:18.221Z  GET    /nope        404  0 ms

Pass a custom format function for total control. The server wraps res.writeHead to capture the actual status code (which may have been set by @ResponseStatus, by HttpError, or by the framework itself for 404/405) and logs once the response ends.

Routes introspection

app.exposeRoutes() serves a JSON snapshot of every registered route at GET /__annotify/routes by default. The router's listRoutes() method walks the trie and returns:

{
  "routes": [
    {
      "method": "GET",
      "path": "/users/:id",
      "handlerName": "get",
      "paramTypes": [{ "kind": "path", "name": "id" }],
      "classCors": {
        "origins": ["http://localhost:5173"],
        "credentials": true,
        "maxAge": 3600
      }
    },
    {
      "method": "POST",
      "path": "/users/",
      "handlerName": "create",
      "paramTypes": [
        { "kind": "body" },
        { "kind": "header", "name": "X-Token" }
      ],
      "statusCode": 201,
      "classCors": { "origins": ["http://localhost:5173"], "credentials": true, "maxAge": 3600 }
    }
  ]
}

Each entry includes method, path, handlerName, paramTypes (with each parameter's kind/name/defaultValue), optional statusCode (from @ResponseStatus), and CORS config (cors for method-level, classCors for class-level). Disable with app.exposeRoutes(null) or change the path with app.exposeRoutes('/api/__routes').

Cluster mode

For production throughput, app.cluster({ port, workers: 'auto' }) forks N workers that share the listening socket. On Linux with SO_REUSEPORT, this scales linearly with the number of CPU cores.

import { AppBuilder, bootstrap } from 'annotify';
import { fileURLToPath } from 'node:url';
import { dirname, resolve } from 'node:path';

const __dirname = dirname(fileURLToPath(import.meta.url));

const app = new AppBuilder();
app.useLogger({ enabled: true });
app.exposeRoutes();
await app.scan(resolve(__dirname, 'controllers'));

// One line — spawns N workers (one per core by default).
await bootstrap(app, { port: 3000, workers: 'auto' });

Or use the lower-level app.cluster({ port, workers }) directly. Behavior:

  • Primary process forks workers and waits for them to listen. SIGINT/SIGTERM on the primary are forwarded to all workers; dying workers are auto-restarted.
  • Worker processes re-enter app.cluster() and call app.listen() instead of forking — they don't fork recursively.
  • Single-process mode: app.listen(port) is unchanged and still works the same way as before.

Performance notes. On Linux with SO_REUSEPORT, expect roughly linear scaling with worker count — 4 cores ≈ 4× the per-process throughput. On Windows, cluster is less efficient because the OS doesn't share listening sockets the same way; use single-process or move to Linux for production. The framework's request hot-path is fast-pathed when the logger is off, and string/Buffer return values bypass JSON.stringify.
