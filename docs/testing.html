Smoke tests

End-to-end checks for every feature annotify exposes. The driver is smoke-simple.ps1; the assertions are status codes + body substrings.

How to run

Make sure the server is up (npm start) in one terminal, then in another:

powershell -ExecutionPolicy Bypass -File smoke-simple.ps1

All 15 checks should report [PASS] and the script ends with:

=== All 15 smoke tests passed ===

The 15 checks

GET /users/?limit=1 200 PASS

Demonstrates: @RequestParam with default value, response shape from handler return value.

GET /users/1 200 PASS

Demonstrates: @PathVariable binding, single-resource fetch.

POST /users/ 200 PASS

Demonstrates: @RequestBody + @RequestHeader working together on the same handler.

PUT /users/3 200 PASS

Demonstrates: @PathVariable + @RequestBody on the same handler.

DEL /users/3 200 PASS

Demonstrates: @DeleteMapping, @PathVariable, return value.

GET /nope 404 PASS

Demonstrates: 404 Not Found response, JSON error envelope { error: "Not Found", message: ... }.

POST /users/1 405 PASS

Demonstrates: 405 Method Not Allowed with Allow: GET, PUT, DELETE header.

GET /api/v1/products 200 PASS

Demonstrates: class-level path prefix. Returns the product array.

GET /api/v1/products/2/reviews/99 200 PASS

Demonstrates: multiple @PathVariable on the same route, deep nesting, literal-and-variable segment mix.

POST /users/ 400 PASS

Demonstrates: HttpError — POST without X-Token header. Handler throws HttpError(400, 'X-Token header is required'), which the server serializes into a 400 JSON response.

CORS (tests 11–13)

GET /users/ (OPTIONS preflight from allowed origin) 204 PASS

Demonstrates: auto-reply to OPTIONS preflight, with Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Credentials: true, and Access-Control-Max-Age: 3600 set from the class-level @CrossOrigin.

GET /users/ (OPTIONS preflight from disallowed origin) 204 PASS

Demonstrates: a preflight from a non-allowlisted origin still returns 204 but no Access-Control-Allow-Origin header — the browser will block the request.

GET /users/1 (with Origin header) 200 PASS

Demonstrates: CORS headers (including Access-Control-Allow-Origin echoing the request origin) on a normal successful response.

Introspection & logging (tests 14–15)

GET /__annotify/routes 200 PASS

Demonstrates: the routes introspection endpoint (app.exposeRoutes()) returns a JSON snapshot of every registered route, including handlerName, paramTypes, statusCode (from @ResponseStatus), and per-method + class-level CORS config.

GET /__annotify/routes (param metadata + CORS) 200 PASS

Demonstrates: the same response contains per-parameter metadata (kind, name, defaultValue) and the configured classCors object. The request logger also writes one access-log line per request — see the server stdout.

How the script works

The script bypasses PowerShell's Invoke-WebRequest (which mishandles 4xx responses on Windows) and uses System.Net.HttpWebRequest directly, catching WebException to read the response body from the exception object.

$req = [System.Net.HttpWebRequest]::Create($Uri)
$req.Method = $Method
try {
  $resp = $req.GetResponse()
  # 2xx path
} catch [System.Net.WebException] {
  $ex = $_.Exception.Response
  # 4xx/5xx path — body is on the exception
}

Each test asserts on the status code AND a substring in the response body. We pick a substring to assert (e.g. "id":1) rather than the full body so tests stay robust against minor formatting changes.
