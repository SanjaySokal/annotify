Errors & status codes

How annotify turns handler outcomes into HTTP responses, and how to surface typed errors from your code.

HttpError

The HttpError class is the public error type. Throw it from a handler to set the response status code and a JSON envelope.

class HttpError extends Error {
  status: number;
  details?: unknown;
  constructor(status: number, message: string, details?: unknown);
}

The server serializes it as:

{
  "error": "<status name>",          // e.g. "Not Found"
  "message": "<your message>",
  "details": <optional payload>
}

Example:

import { HttpError } from 'annotify';

@GetMapping('/:id')
get(@PathVariable('id') id: string) {
  const user = users.find(u => u.id === Number(id));
  if (!user) throw new HttpError(404, `User ${id} not found`);
  return user;
}
$ curl http://localhost:3000/users/999
{
  "error": "Not Found",
  "message": "User 999 not found"
}

All status codes annotify produces

SituationStatusBodyNotes
Handler returns a value200JSON.stringify(value)Default success path.
Handler returns undefined204emptyUse when you've already serialized manually or there's nothing to say.
Handler throws HttpError(s, msg)s{ error, message, details? }Use this for everything user-facing.
Handler throws anything else500{ error: "Internal Server Error", message }Catch and rethrow as HttpError if you want a different code.
Path doesn't exist404{ error: "Not Found", message }Returned by the server before handler dispatch.
Path exists, wrong method405 + Allow: GET, PUT, …{ error: "Method Not Allowed", message }The Allow header lists valid methods.
Body > 1 MB413{ error: "Payload Too Large", message }Body cap is hard-coded at 1 MB.
Body content-type ≠ JSON415{ error: "Unsupported Media Type", message }Returned when @RequestBody is used but the request isn't application/json.

Custom error subclasses

Extend HttpError for application-specific error types:

import { HttpError } from 'annotify';

export class NotFoundError extends HttpError {
  constructor(what: string) {
    super(404, `${what} not found`);
  }
}

export class AuthError extends HttpError {
  constructor(msg = 'Unauthorized') {
    super(401, msg);
  }
}
@GetMapping('/:id')
get(@PathVariable('id') id: string) {
  const user = users.find(u => u.id === Number(id));
  if (!user) throw new NotFoundError(`User ${id}`);
  return user;
}

Handler already wrote to res?

If your handler writes to res directly (for streaming, SSE, file downloads), annotify will not write a second response. Return undefined or throw nothing — the framework checks res.writableEnded after the handler returns.

@GetMapping('/download')
download(req: IncomingMessage, res: ServerResponse) {
  res.setHeader('Content-Type', 'application/octet-stream');
  res.setHeader('Content-Disposition', 'attachment; filename="data.bin"');
  res.end(Buffer.from('…'));
  // returning undefined → annotify sees writableEnded, skips its 204 path
}

Global error filtering

There's no built-in @ExceptionHandler or @ControllerAdvice yet — wrap async handler logic in a try/catch inside the handler if you want to map domain errors to HTTP status codes:

@GetMapping('/:id')
async get(@PathVariable('id') id: string) {
  try {
    return await this.users.findById(id);
  } catch (err) {
    if (err.code === 'TIMEOUT') throw new HttpError(504, 'Upstream timeout');
    throw new HttpError(500, 'Internal error');
  }
}

Future work. An @ExceptionHandler decorator is a natural next addition — it would let you declare a method as a catch-all for one or more exception types and let the framework route exceptions through it.
